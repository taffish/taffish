# 模块文档模板

每个重要目录或文件都应该按统一逻辑写文档。重点不是形式整齐，而是让维护者快速建立正确心智模型。

## 推荐模板

```markdown
# 模块名

## 作用

说明这部分代码为什么存在，它解决什么问题，不解决什么问题。

## 系统位置

说明它处在 TAFFISH 的哪一层，上游是谁，下游是谁。

## 上下游交互

说明它接收什么结构，返回什么结构，调用哪些模块，被哪些模块调用。

## 核心数据结构或不变量

列出维护时不能破坏的规则。比如字段含义、路径约定、错误类型、返回值形状、加载顺序等。

## 公开 API

列出外部模块应该调用的函数、宏、变量或类。只写真正稳定或半稳定的接口。

## 实现要点

解释容易误解的实现细节。这里不需要逐行解释代码，而要解释为什么这么写。

## 修改指南

告诉后来者改这里时要同步检查哪里、容易引入什么问题、应该增加什么测试或文档。
```

## 写作要求

每篇模块文档至少要回答四个问题：

1. 这部分代码的职责边界是什么？
2. 它在完整流程中处于什么位置？
3. 它依赖哪些输入契约，又向下游承诺哪些输出契约？
4. 修改它时最容易破坏什么？

## 不推荐的写法

不推荐只写函数列表：

```markdown
## API

- `foo`
- `bar`
- `baz`
```

这种写法对维护帮助很小，因为后来者仍然不知道为什么需要这些函数、何时调用、调用前后系统状态如何变化。

## 推荐的写法

更好的写法是先解释模块意义：

```markdown
## 作用

`binder` 是 parser 和 compiler 之间的语义绑定层。parser 只理解源码结构，binder 负责把外部输入参数和运行上下文绑定到 program，形成 compiler 可以直接消费的 result。

## 不变量

- parser 不读取用户实际输入值。
- binder 不重新解释 TAF 源码语法。
- compiler 不直接读取 CLI 参数。
```

这类内容能帮助维护者判断一个新功能应该加在哪里。

# TAF 语言契约

本页记录 TAF 语言在 `taffish-core` 中应保持的核心契约。更完整的语言说明见 [TAF 语言规范草案](taf-language-spec.md)。本页保留为维护编译器时的高层不变量清单。

## 文件角色

TAF 文件描述一个可编译的工具或流程。它既要能表达参数，也要能表达运行块，还要能通过标签选择不同执行模型。

## 基本结构

当前 TAF 编译器识别：

1. 空行。
2. 注释。
3. 标签。
4. 普通代码。
5. `ARGS` 块。
6. `RUN` 块。
7. 子标签。

parser 可以把某些裸代码规范化为默认运行块，这是为了让简单 TAF 更易写。但这种便利不应该破坏结构清晰性。

## ARGS 契约

`ARGS` 描述用户输入参数。它最终会被转为 `han.args` 参数规格。

维护时应保持：

1. parser 负责把 ARGS 转换为参数规格。
2. binder 负责把真实输入绑定到参数规格。
3. compiler 不直接读取原始 CLI 参数。

## RUN 契约

`RUN` 描述执行逻辑。它可以通过标签交给不同 emitter。

维护时应保持：

1. RUN 块的结构由 parser 决定。
2. RUN 块的语义发射由 emitter 决定。
3. compiler 只组织流程，不写死具体标签行为。

## 位置与错误

lexer 应保留足够位置信息，使 parser、binder 或 compiler 抛错时能追踪到源文件位置。TAFFISH 面向可复现工具和流程，错误定位能力是语言体验的一部分。

## 未稳定部分

以下内容如果要变化，应先补充更完整的标准：

1. 子标签组合规则。
2. 内联 taffish 组合语法。
3. taf-app 命令模式细节。
4. container 标签的完整参数语法。

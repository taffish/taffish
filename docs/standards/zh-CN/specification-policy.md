# 规范定位与版本策略

本页说明 TAFFISH 规范文档的定位。当前文档应被理解为 `TAFFISH Specification Draft v0.1`，中文可称为“TAFFISH 规范草案 v0.1”。

## 为什么是规范草案

TAFFISH 现在仍以一个参考实现为中心。语言、项目格式、hub index、安装元数据和容器运行规则都已经有了清晰轮廓，但生态还没有发展到“多个实现之间需要外部仲裁”的阶段。

因此，当前不应把 TAFFISH 宣称为外部正式标准。更合适的定位是：

1. 参考实现定义当前事实行为。
2. 规范草案把这些事实行为提升为可讨论、可维护、可兼容演进的契约。
3. 未来的一致性测试验证实现是否符合这些契约。
4. 当生态成熟后，再考虑正式标准化、治理委员会或 TEP 流程。

## 与正式标准的区别

正式标准通常需要多个实现、外部利益相关者、稳定提案流程和兼容性裁决机制。TAFFISH 当前更需要快速迭代和自我修正。

规范草案不要求冻结所有设计。它要求每一次变化都能回答：

1. 这会不会破坏已有 `.taf` 文件？
2. 这会不会破坏已有 taf-app 项目？
3. 这会不会破坏已发布的 hub index？
4. 这会不会破坏用户已经安装的命令？
5. 如果会破坏，迁移路径是什么？

## 规范版本

规范文档本身使用草案版本：

| 版本 | 含义 |
| --- | --- |
| `Draft v0.1` | 与当前 TAFFISH 0.3.x 参考实现对齐，允许修正和补充。 |
| `Draft v0.2` | 当 taffish-hub 迁移完成、更多 taf-app 被实际发布后更新。 |
| `v1.0` | 当语言、项目格式、hub index 和安装格式经过实际生态验证后再考虑。 |

schema 字段使用独立版本，例如 `taffish.index/v1`、`taffish.install/v1`、`taffish.config/v1`。schema 版本不必与 TAFFISH 程序版本完全一致。

## 规范关键词

本文档中的关键词约定如下：

| 关键词 | 含义 |
| --- | --- |
| 必须 | 参考实现和兼容实现都应遵守。 |
| 应该 | 强烈建议遵守，除非有明确兼容理由。 |
| 可以 | 允许但不强制。 |
| 当前实现 | 描述现有代码行为，不一定已经承诺长期稳定。 |
| 保留 | 不建议外部依赖，未来可能变更或补齐。 |

## 规范性文本与说明性文本

规范文档包含两类文本：

| 类型 | 作用 | 例子 |
| --- | --- | --- |
| 规范性文本 | 定义兼容实现必须或应该遵守的行为。 | schema 名称、字段类型、命名规则、解析优先级。 |
| 说明性文本 | 解释为什么这样设计，或说明当前参考实现如何做到。 | 某个 Common Lisp 函数名、当前实现的内部数据结构。 |

未来如果出现第二个 TAFFISH 实现，它需要遵守规范性文本，但不需要复刻当前 Common Lisp 实现的内部结构。

## 成熟度标签

每个规范主题可以使用以下成熟度：

| 标签 | 含义 |
| --- | --- |
| Draft | 已写入规范草案，但仍允许根据实现和生态反馈调整。 |
| Stable | 已被参考实现和实际 taf-app 反复验证，破坏性改变需要迁移计划。 |
| Experimental | 可以试用，但不建议发布生态依赖。 |
| Deprecated | 仍可读取或运行，但不建议继续生成。 |
| Reserved | 为未来设计保留，当前不应依赖。 |

TAFFISH 0.3.x 阶段大多数文档属于 `Draft v0.1`。`taffish-hub` 迁移完成后，可以把其中一部分提升到 `Draft v0.2` 或 `Stable`。

## 参考实现

当前参考实现是本仓库中的 Common Lisp 实现，包括：

1. `taffish-core`：TAF 语言到 shell 的编译核心。
2. `taf-core`：项目、hub、安装、配置、history 和诊断工具。
3. `taffish-cli` 与 `taf-cli`：命令行入口。
4. `vendor/han`：TAFFISH 使用的基础库。

当规范草案与代码冲突时，短期以内以当前代码作为事实行为；长期应该修正文档或代码，使二者重新一致。

## 不稳定区域

以下区域仍处于较强演进期：

1. `taffish` 内联流程引用语法。
2. `taf-app` 命令模式的细节。
3. hub index 的完整生产流程。
4. Gitee 镜像策略与多源同步策略。
5. 容器镜像缓存、SIF 命名和高级后端参数。
6. 一致性测试集合。

这些区域可以记录现状，但不应过早冻结成不可变标准。

# TAFFISH MCP Interface Specification

This page defines the stable contract for the `taffish-mcp` stdio MCP server. It is a consumer-facing AI integration interface, not a replacement for `taf` or `taffish`.

## Scope

Normative: `taffish-mcp` exposes safe structured access to TAFFISH through MCP tools, resources, and prompts.

The interface MAY expose:

1. Version and help metadata.
2. Source/file validation, summary, and compile-to-shell operations.
3. Installed taf-app lookup, inspection, usage summary, trust/smoke metadata exposure, and invocation compile.
4. Current project check, inspection, usage summary, trust/smoke metadata exposure, compile, and safe build planning.
5. Hub search/info/list/which/update operations.
6. Config, environment, and history query operations.
7. Dry-run install/uninstall helpers.

The interface MUST NOT expose:

1. Arbitrary shell execution.
2. `taf run`.
3. `taf publish`.
4. Container image build, push, or registry mutation.
5. Any operation that silently overwrites user files without an explicit force-like argument.

## Transport

Normative: the primary transport is stdio JSON-RPC as expected by MCP clients.

`stdout` is reserved for MCP protocol messages. Diagnostic logs should go to `stderr` so they do not corrupt JSON-RPC framing.

## Tool Naming

Normative: tool names use the `taffish_` prefix.

Names should be descriptive and stable. If a tool is renamed, the old name should either remain as an alias until a documented removal point, or the change should be treated as an MCP interface breaking change.

Recommended domains:

| Domain | Examples |
| --- | --- |
| compiler | `taffish_validate_source`, `taffish_compile_file` |
| taf-app | `taffish_inspect_app`, `taffish_compile_app_invocation` |
| project | `taffish_check_project`, `taffish_compile_project` |
| Hub | `taffish_search_apps`, `taffish_install_app` |
| system | `taffish_check_environment`, `taffish_get_config` |

## Structured Results

Normative: successful operations that return structured data should include enough fields for an AI client to proceed without parsing prose.

Normative: expected business failures should prefer this shape:

```json
{
  "ok": false,
  "error": {
    "kind": "error-kind",
    "message": "human-readable message"
  }
}
```

`kind` should be stable enough for client-side branching. `message` is for human explanation and debugging.

Compile operations that succeed SHOULD include `ok: true`, generated `shell` when applicable, and summary fields such as byte count or validation metadata.

## Compile Safety

Normative: source, file, project, and app invocation compile tools may return shell code, but MUST NOT execute it.

App invocation compile should validate arguments using the same underlying argument-binding logic as normal TAFFISH compilation. The result is a safer preview path for AI clients, not a runtime.

Container backend availability may be detected and included in compiler context, but MCP compile operations must not pull images or start containers.

For tools that accept `containerBackend`, the effective forced backend priority is:

1. explicit `containerBackend` tool argument.
2. `TAFFISH_CONTAINER_BACKEND=apptainer|podman|docker` from the MCP server environment.
3. no forced backend.

The forced backend only applies to generic `<container:...>` tags in generated shell. Explicit `<docker:...>`, `<podman:...>`, and `<apptainer:...>` tags remain explicit.

## Smoke And Trust Metadata

Normative: app and project inspection/summary tools SHOULD expose smoke/trust
metadata when available.

For containerized projects, `taf check` validates smoke metadata and rejects
default TODO placeholders. MCP may report this validation result, but MUST NOT
execute smoke commands, pull images, or start containers to verify smoke.

For indexed apps, trust metadata may include fields such as container digest,
supported platforms, smoke status, and source commit. These fields are data for
AI reasoning and audit preparation; they are not proof generated by MCP itself.

## Resources

Resources provide reference material to the AI client. They should remain compact, stable, and actionable.

Recommended resources:

1. Tool overview.
2. Compiler usage model.
3. Hub/install usage model.
4. Project inspection model.
5. Current project files when a project root is available.

Resources should not duplicate the whole codebase or become hidden prompts for unsafe operations.

## Prompts

Prompts encode recommended workflows for AI clients. They should guide the model toward safe inspection before action.

Examples:

1. Inspect a project before changing or building it.
2. Resolve and inspect a taf-app before compiling an invocation.
3. Validate source before showing generated shell.

Prompts should not encourage execution, publish, or image mutation through MCP.

## Compatibility

MCP interface compatibility is separate from TAFFISH CLI compatibility. A change can be CLI-compatible but MCP-breaking if it changes tool names, required arguments, result shape, or resource URIs.

Before changing the MCP interface, check:

1. Are existing tool names preserved?
2. Are required arguments still accepted?
3. Are successful and failed results still structurally readable?
4. Are resources and prompts still concise and aligned with the safety boundary?

# Generated Shell Contract

One key TAFFISH design choice is compiling TAF into shell. Generated shell is TAFFISH's runtime boundary and the foundation of portability.

## Basic Goals

Generated shell should be:

1. Readable.
2. Executable.
3. Debuggable.
4. Composable.
5. As POSIX-shell-compatible as practical.

## Compilation Output Structure

`compiler.lisp` organizes:

1. shebang.
2. prelude.
3. emitter output for each block.
4. finalize.

Specific tag shell fragments are generated by emitters.

## Quoting And Paths

The most fragile parts of shell output are quoting, paths, and spaces. Be especially careful with:

1. user input arguments.
2. working directory.
3. home directory.
4. container mount paths.
5. temporary script paths.
6. arguments passed to Docker, Podman, or Apptainer.

Values from users or external environment should not be directly concatenated into shell.

## Container Runtime Contract

The container emitter is one of the most complex generated-shell areas. It coordinates:

1. backend selection.
2. image or SIF.
3. working directory.
4. mounts.
5. heredoc or single-command execution.
6. backend differences.

When changing container output, consider Docker, Podman, and Apptainer together.

## Debuggability

Generated shell should not be over-compressed. One advantage of TAFFISH is that compiled output can be read and debugged by humans. Sacrificing readability for shorter output is usually not worth it.
